010/**
011 * Object relational query for finding a List, Set, Map or single entity bean.
012 * <p>
013 * Example: Create the query using the API.
014 * </p>
015 * 
016 * <pre>{@code
017 *
018 * List<Order> orderList = 
019 *   ebeanServer.find(Order.class)
020 *     .fetch("customer")
021 *     .fetch("details")
022 *     .where()
023 *       .like("customer.name","rob%")
024 *       .gt("orderDate",lastWeek)
025 *     .orderBy("customer.id, id desc")
026 *     .setMaxRows(50)
027 *     .findList();
028 *   
029 * ...
030 * }</pre>
031 * 
032 * <p>
033 * Example: The same query using the query language
034 * </p>
035 * 
036 * <pre>{@code
037 *
038 * String oql = 
039 *      "  find  order "
040 *      +" fetch customer "
041 *      +" fetch details "
042 *      +" where customer.name like :custName and orderDate > :minOrderDate "
043 *      +" order by customer.id, id desc "
044 *      +" limit 50 ";
045 *   
046 * Query<Order> query = ebeanServer.createQuery(Order.class, oql);
047 * query.setParameter("custName", "Rob%");
048 * query.setParameter("minOrderDate", lastWeek);
049 *   
050 * List<Order> orderList = query.findList();
051 * ...
052 * }</pre>
053 * 
054 * <p>
055 * Example: Using a named query called "with.cust.and.details"
056 * </p>
057 * 
058 * <pre>{@code
059 *
060 * Query<Order> query = ebeanServer.createNamedQuery(Order.class,"with.cust.and.details");
061 * query.setParameter("custName", "Rob%");
062 * query.setParameter("minOrderDate", lastWeek);
063 *   
064 * List<Order> orderList = query.findList();
065 * ...
066 * }</pre>
067 * 
068 * <h3>Autofetch</h3>
069 * <p>
070 * Ebean has built in support for "Autofetch". This is a mechanism where a query
071 * can be automatically tuned based on profiling information that is collected.
072 * </p>
073 * <p>
074 * This is effectively the same as automatically using select() and fetch() to
075 * build a query that will fetch all the data required by the application and no
076 * more.
077 * </p>
078 * <p>
079 * It is expected that Autofetch will be the default approach for many queries
080 * in a system. It is possibly not as useful where the result of a query is sent
081 * to a remote client or where there is some requirement for "Read Consistency"
082 * guarantees.
083 * </p>
084 * 
085 * <h3>Query Language</h3>
086 * <p>
087 * <b>Partial Objects</b>
088 * </p>
089 * <p>
090 * The <em>find</em> and <em>fetch</em> clauses support specifying a list of
091 * properties to fetch. This results in objects that are "partially populated".
092 * If you try to get a property that was not populated a "lazy loading" query
093 * will automatically fire and load the rest of the properties of the bean (This
094 * is very similar behaviour as a reference object being "lazy loaded").
095 * </p>
096 * <p>
097 * Partial objects can be saved just like fully populated objects. If you do
098 * this you should remember to include the <em>"Version"</em> property in the
099 * initial fetch. If you do not include a version property then optimistic
100 * concurrency checking will occur but only include the fetched properties.
101 * Refer to "ALL Properties/Columns" mode of Optimistic Concurrency checking.
102 * </p>
103 * 
104 * <pre>{@code
105 * [ find  {bean type} [ ( * | {fetch properties} ) ] ]
106 * [ fetch {associated bean} [ ( * | {fetch properties} ) ] ]
107 * [ where {predicates} ]
108 * [ order by {order by properties} ]
109 * [ limit {max rows} [ offset {first row} ] ]
110 * }</pre>
111 * 
112 * <p>
113 * <b>FIND</b> <b>{bean type}</b> [ ( <i>*</i> | <i>{fetch properties}</i> ) ]
114 * </p>
115 * <p>
116 * With the find you specify the type of beans to fetch. You can optionally
117 * specify a list of properties to fetch. If you do not specify a list of
118 * properties ALL the properties for those beans are fetched.
119 * </p>
120 * <p>
121 * In object graph terms the <em>find</em> clause specifies the type of bean at
122 * the root level and the <em>fetch</em> clauses specify the paths of the object
123 * graph to populate.
124 * </p>
125 * <p>
126 * <b>FETCH</b> <b>{associated property}</b> [ ( <i>*</i> | <i>{fetch
127 * properties}</i> ) ]
128 * </p>
129 * <p>
130 * With the fetch you specify the associated property to fetch and populate. The
131 * associated property is a OneToOnem, ManyToOne, OneToMany or ManyToMany
132 * property. When the query is executed Ebean will fetch the associated data.
133 * </p>
134 * <p>
135 * For fetch of a path we can optionally specify a list of properties to fetch.
136 * If you do not specify a list of properties ALL the properties for that bean
137 * type are fetched.
138 * </p>
139 * <p>
140 * <b>WHERE</b> <b>{list of predicates}</b>
141 * </p>
142 * <p>
143 * The list of predicates which are joined by AND OR NOT ( and ). They can
144 * include named (or positioned) bind parameters. These parameters will need to
145 * be bound by {@link Query#setParameter(String, Object)}.
146 * </p>
147 * <p>
148 * <b>ORDER BY</b> <b>{order by properties}</b>
149 * </p>
150 * <p>
151 * The list of properties to order the result. You can include ASC (ascending)
152 * and DESC (descending) in the order by clause.
153 * </p>
154 * <p>
155 * <b>LIMIT</b> <b>{max rows}</b> [ OFFSET <i>{first row}</i> ]
156 * </p>
157 * <p>
158 * The limit offset specifies the max rows and first row to fetch. The offset is
159 * optional.
160 * </p>
161 * <h4>Examples of Ebean's Query Language</h4>
162 * <p>
163 * Find orders fetching all its properties
164 * </p>
165 * 
166 * <pre>{@code
167 * find order
168 * }</pre>
169 * 
170 * <p>
171 * Find orders fetching all its properties
172 * </p>
173 * 
174 * <pre>{@code
175 * find order (*)
176 * }</pre>
177 * 
178 * <p>
179 * Find orders fetching its id, shipDate and status properties. Note that the id
180 * property is always fetched even if it is not included in the list of fetch
181 * properties.
182 * </p>
183 * 
184 * <pre>{@code
185 * find order (shipDate, status)
186 * }</pre>
187 * 
188 * <p>
189 * Find orders with a named bind variable (that will need to be bound via
190 * {@link Query#setParameter(String, Object)}).
191 * </p>
192 * 
193 * <pre>{@code
194 * find order
195 * where customer.name like :custLike
196 * }</pre>
197 * 
198 * <p>
199 * Find orders and also fetch the customer with a named bind parameter. This
200 * will fetch and populate both the order and customer objects.
201 * </p>
202 * 
203 * <pre>{@code
204 * find  order
205 * fetch customer
206 * where customer.id = :custId
207 * }</pre>
208 * 
209 * <p>
210 * Find orders and also fetch the customer, customer shippingAddress, order
211 * details and related product. Note that customer and product objects will be
212 * "Partial Objects" with only some of their properties populated. The customer
213 * objects will have their id, name and shipping address populated. The product
214 * objects (associated with each order detail) will have their id, sku and name
215 * populated.
216 * </p>
217 * 
218 * <pre>{@code
219 * find  order
220 * fetch customer (name)
221 * fetch customer.shippingAddress
222 * fetch details
223 * fetch details.product (sku, name)
224 * }</pre>
225 * 
226 * <h3>Early parsing of the Query</h3>
227 * <p>
228 * When you get a Query object from a named query, the query statement has
229 * already been parsed. You can then add to that query (add fetch paths, add to
230 * the where clause) or override some of its settings (override the order by
231 * clause, first rows, max rows).
232 * </p>
233 * <p>
234 * The thought is that you can use named queries as a 'starting point' and then
235 * modify the query to suit specific needs.
236 * </p>
237 * <h3>Building the Where clause</h3>
238 * <p>
239 * You can add to the where clause using Expression objects or a simple String.
240 * Note that the ExpressionList has methods to add most of the common
241 * expressions that you will need.
242 * <ul>
243 * <li>where(String addToWhereClause)</li>
244 * <li>where().add(Expression expression)</li>
245 * <li>where().eq(propertyName, value).like(propertyName , value)...</li>
246 * </ul>
247 * </p>
248 * <p>
249 * The full WHERE clause is constructed by appending together
250 * <li>original query where clause (Named query or query.setQuery(String oql))</li>
251 * <li>clauses added via query.where(String addToWhereClause)</li>
252 * <li>clauses added by Expression objects</li>
253 * </p>
254 * <p>
255 * The above is the order that these are clauses are appended to give the full
256 * WHERE clause.
257 * </p>
258 * <h3>Design Goal</h3>
259 * <p>
260 * This query language is NOT designed to be a replacement for SQL. It is
261 * designed to be a simple way to describe the "Object Graph" you want Ebean to
262 * build for you. Each find/fetch represents a node in that "Object Graph" which
263 * makes it easy to define for each node which properties you want to fetch.
264 * </p>
265 * <p>
266 * Once you hit the limits of this language such as wanting aggregate functions
267 * (sum, average, min etc) or recursive queries etc you use SQL. Ebean's goal is
268 * to make it as easy as possible to use your own SQL to populate entity beans.
269 * Refer to {@link RawSql} .
270 * </p>
271 * 
272 * @param <T>
273 *          the type of Entity bean this query will fetch.
274 */

312   * <p/>
313   * Note that the same persistence Context is used for subsequent lazy loading and query join queries.

315   * Note that #findEach uses a 'per object graph' PersistenceContext so this scope is ignored for

334   * <p>
335   * If you do not call this method on a query the "Implicit Autofetch mode" is
336   * used to determine if Autofetch should be used for a given query.
337   * </p>
338   * <p>
339   * Autofetch can add additional fetch paths to the query and specify which

347   * Set the default lazy loading batch size to use.
348   * <p>
349   * When lazy loading is invoked on beans loaded by this query then this sets the

352   * @param lazyLoadBatchSize the number of beans to lazy load in a single batch
353   */
354  public Query<T> setLazyLoadBatchSize(int lazyLoadBatchSize);
355
356  /**
357   * Explicitly set a comma delimited list of the properties to fetch on the
358   * 'main' root level entity bean (aka partial object). Note that '*' means all
359   * properties.
360   * <p>
361   * You use {@link #fetch(String, String)} to specify specific properties to fetch
362   * on other non-root level paths of the object graph.
363   * </p>
364   *
365   * <pre>{@code
366   *
367   * List<Customer> customers =
368   *     ebeanServer.find(Customer.class)
369   *     // Only fetch the customer id, name and status.
370   *     // This is described as a "Partial Object"
371   *     .select("name, status")
372   *     .where.ilike("name", "rob%")
373   *     .findList();
374   *
375   * }</pre>
376   *
377   * @param fetchProperties
378   *          the properties to fetch for this bean (* = all properties).
379   */
380  public Query<T> select(String fetchProperties);
381
382  /**
383   * Specify a path to <em>fetch</em> with its specific properties to include
384   * (aka partial object).
385   * <p>
386   * When you specify a join this means that property (associated bean(s)) will
387   * be fetched and populated. If you specify "*" then all the properties of the
388   * associated bean will be fetched and populated. You can specify a comma
389   * delimited list of the properties of that associated bean which means that
390   * only those properties are fetched and populated resulting in a
391   * "Partial Object" - a bean that only has some of its properties populated.
392   * </p>
393   * 
394   * <pre>{@code
395   *
396   * // query orders...
397   * List<Order> orders =
398   *     ebeanserver.find(Order.class)
399   *       // fetch the customer...
400   *       // ... getting the customers name and phone number
401   *       .fetch("customer", "name, phoneNumber")
402   * 
403   *       // ... also fetch the customers billing address (* = all properties)
404   *       .fetch("customer.billingAddress", "*")
405   *       .findList();
406   * }</pre>
407   * 
408   * <p>
409   * If columns is null or "*" then all columns/properties for that path are
410   * fetched.
411   * </p>
412   * 
413   * <pre>{@code
414   *
415   * // fetch customers (their id, name and status)
416   * List<Customer> customers =
417   *     ebeanServer.find(Customer.class)
418   *     .select("name, status")
419   *     .fetch("contacts", "firstName,lastName,email")
420   *     .findList();
421   *
422   * }</pre>
423   * 
424   * @param path
425   *          the path of an associated (1-1,1-M,M-1,M-M) bean.
426   * @param fetchProperties
427   *          properties of the associated bean that you want to include in the
428   *          fetch (* means all properties, null also means all properties).
429   */
430  public Query<T> fetch(String path, String fetchProperties);
431
432  /**
433   * Additionally specify a FetchConfig to use a separate query or lazy loading
434   * to load this path.
435   *
436   * <pre>{@code
437   *
438   * // fetch customers (their id, name and status)
439   * List<Customer> customers =
440   *     ebeanServer.find(Customer.class)
441   *     .select("name, status")
442   *     .fetch("contacts", "firstName,lastName,email", new FetchConfig().lazy(10))
443   *     .findList();
444   *
445   * }</pre>
446   */
447  public Query<T> fetch(String assocProperty, String fetchProperties, FetchConfig fetchConfig);
448
449  /**
450   * Specify a path to load including all its properties.
451   * <p>
452   * The same as {@link #fetch(String, String)} with the fetchProperties as "*".
453   * </p>
454   * <pre>{@code
455   *
456   * // fetch customers (their id, name and status)
457   * List<Customer> customers =
458   *     ebeanServer.find(Customer.class)
459   *     // eager fetch the contacts
460   *     .fetch("contacts")
461   *     .findList();
462   *
463   * }</pre>
464   *
465   * @param path
466   *          the property of an associated (1-1,1-M,M-1,M-M) bean.
467   */
468  public Query<T> fetch(String path);
469
470  /**
471   * Additionally specify a JoinConfig to specify a "query join" and or define
472   * the lazy loading query.
473   *
474   *
475   * <pre>{@code
476   *
477   * // fetch customers (their id, name and status)
478   * List<Customer> customers =
479   *     ebeanServer.find(Customer.class)
480   *     // lazy fetch contacts with a batch size of 100
481   *     .fetch("contacts", new FetchConfig().lazy(100))
482   *     .findList();
483   *
484   * }</pre>
485   */
486  public Query<T> fetch(String path, FetchConfig joinConfig);

537  /**
538   * Execute the query processing the beans one at a time.
539   * <p>
540   * This method is appropriate to process very large query results as the
541   * beans are consumed one at a time and do not need to be held in memory
542   * (unlike #findList #findSet etc)
543   * </p>
544   * <p>
545   * Note that internally Ebean can inform the JDBC driver that it is expecting larger
546   * resultSet and specifically for MySQL this hint is required to stop it's JDBC driver
547   * from buffering the entire resultSet. As such, for smaller resultSets findList() is
548   * generally preferable.
549   * </p>
550   * <p>
551   * Compared with #findEachWhile this will always process all the beans where as
552   * #findEachWhile provides a way to stop processing the query result early before
553   * all the beans have been read.
554   * </p>
555   * <p>
556   * This method is functionally equivalent to findIterate() but instead of using an
557   * iterator uses the QueryEachConsumer (SAM) interface which is better suited to use
558   * with Java8 closures.
559   * </p>
560   *
561   * <pre>{@code
562   *
563   *  ebeanServer.find(Customer.class)
564   *     .where().eq("status", Status.NEW)
565   *     .order().asc("id")
566   *     .findEach((Customer customer) -> {
567   *
568   *       // do something with customer
569   *       System.out.println("-- visit " + customer);
570   *     });
571   *
572   * }</pre>
573   *
574   * @param consumer
575   *          the consumer used to process the queried beans.
576   */
577  public void findEach(QueryEachConsumer<T> consumer);
578
579  /**
580   * Execute the query using callbacks to a visitor to process the resulting
581   * beans one at a time.
582   * <p>
583   * This method is functionally equivalent to findIterate() but instead of using an
584   * iterator uses the QueryEachWhileConsumer (SAM) interface which is better suited to use
585   * with Java8 closures.
586   * </p>
587
588   *
589   * <pre>{@code
590   *
591   *  ebeanServer.find(Customer.class)
592   *     .fetch("contacts", new FetchConfig().query(2))
593   *     .where().eq("status", Status.NEW)
594   *     .order().asc("id")
595   *     .setMaxRows(2000)
596   *     .findEachWhile((Customer customer) -> {
597   *
598   *       // do something with customer
599   *       System.out.println("-- visit " + customer);
600   *
601   *       // return true to continue processing or false to stop
602   *       return (customer.getId() < 40);
603   *     });
604   *
605   * }</pre>
606   *
607   * @param consumer
608   *          the consumer used to process the queried beans.
609   */
610  public void findEachWhile(QueryEachWhileConsumer<T> consumer);

678  /**
679   * Execute the query returning either a single bean or null (if no matching
680   * bean is found).
681   * <p>
682   * If more than 1 row is found for this query then a PersistenceException is
683   * thrown.
684   * </p>
685   * <p>
686   * This is useful when your predicates dictate that your query should only
687   * return 0 or 1 results.
688   * </p>
689   * 
690   * <pre>{@code
691   *
692   * // assuming the sku of products is unique...
693   * Product product =
694   *     ebeanServer.find(Product.class)
695   *         .where().eq("sku", "aa113")
696   *         .findUnique();
697   * ...
698   * }</pre>
699   * 
700   * <p>
701   * It is also useful with finding objects by their id when you want to specify
702   * further join information.
703   * </p>
704   * 
705   * <pre>{@code
706   *
707   * // Fetch order 1 and additionally fetch join its order details...
708   * Order order = 
709   *     ebeanServer.find(Order.class)
710   *       .setId(1)
711   *       .fetch("details")
712   *       .findUnique();
713   *
714   * // the order details were eagerly loaded
