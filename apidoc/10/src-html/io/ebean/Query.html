013/**
014 * Object relational query for finding a List, Set, Map or single entity bean.
015 * <p>
016 * Example: Create the query using the API.
017 * </p>
018 * <p>
019 * <pre>{@code
020 *
021 * List<Order> orderList =
022 *   ebeanServer.find(Order.class)
023 *     .fetch("customer")
024 *     .fetch("details")
025 *     .where()
026 *       .like("customer.name","rob%")
027 *       .gt("orderDate",lastWeek)
028 *     .orderBy("customer.id, id desc")
029 *     .setMaxRows(50)
030 *     .findList();
031 *
032 * ...
033 * }</pre>
034 * <p>
035 * Example: The same query using the query language
036 * </p>
037 * <pre>{@code
038 *
039 * String oql =
040 *      +" fetch customer "
041 *      +" fetch details "
042 *      +" where customer.name like :custName and orderDate > :minOrderDate "
043 *      +" order by customer.id, id desc "
044 *      +" limit 50 ";
045 *
046 * Query<Order> query = ebeanServer.createQuery(Order.class, oql);
047 * query.setParameter("custName", "Rob%");
048 * query.setParameter("minOrderDate", lastWeek);
049 *
050 * List<Order> orderList = query.findList();
051 * ...
052 * }</pre>
053 * <p>
054 * Example: Using a named query called "with.cust.and.details"
055 * </p>
056 * <pre>{@code
057 *
058 * Query<Order> query = ebeanServer.createNamedQuery(Order.class,"with.cust.and.details");
059 * query.setParameter("custName", "Rob%");
060 * query.setParameter("minOrderDate", lastWeek);
061 *
062 * List<Order> orderList = query.findList();
063 * ...
064 * }</pre>
065 * <h3>AutoTune</h3>
066 * <p>
067 * Ebean has built in support for "AutoTune". This is a mechanism where a query
068 * can be automatically tuned based on profiling information that is collected.
069 * </p>
070 * <p>
071 * This is effectively the same as automatically using select() and fetch() to
072 * build a query that will fetch all the data required by the application and no
073 * more.
074 * </p>
075 * <p>
076 * It is expected that AutoTune will be the default approach for many queries
077 * in a system. It is possibly not as useful where the result of a query is sent
078 * to a remote client or where there is some requirement for "Read Consistency"
079 * guarantees.
080 * </p>
081 * <h3>Query Language</h3>
082 * <p>
083 * <b>Partial Objects</b>
084 * </p>
085 * <p>
086 * The <em>find</em> and <em>fetch</em> clauses support specifying a list of
087 * properties to fetch. This results in objects that are "partially populated".
088 * If you try to get a property that was not populated a "lazy loading" query
089 * will automatically fire and load the rest of the properties of the bean (This
090 * is very similar behaviour as a reference object being "lazy loaded").
091 * </p>
092 * <p>
093 * Partial objects can be saved just like fully populated objects. If you do
094 * this you should remember to include the <em>"Version"</em> property in the
095 * initial fetch. If you do not include a version property then optimistic
096 * concurrency checking will occur but only include the fetched properties.
097 * Refer to "ALL Properties/Columns" mode of Optimistic Concurrency checking.
098 * </p>
099 * <pre>{@code
100 * [ select [ ( * | {fetch properties} ) ] ]
101 * [ fetch {path} [ ( * | {fetch properties} ) ] ]
102 * [ where {predicates} ]
103 * [ order by {order by properties} ]
104 * [ limit {max rows} [ offset {first row} ] ]
105 * }</pre>
106 * <p>
107 * <b>SELECT</b> [ ( <i>*</i> | <i>{fetch properties}</i> ) ]
108 * </p>
109 * <p>
110 * With the select you can specify a list of properties to fetch.
111 * </p>
112 * <p>
113 * <b>FETCH</b> <b>{path}</b> [ ( <i>*</i> | <i>{fetch properties}</i> ) ]
114 * </p>
115 * <p>
116 * With the fetch you specify the associated property to fetch and populate. The
117 * path is a OneToOne, ManyToOne, OneToMany or ManyToMany property.
118 * </p>
119 * <p>
120 * For fetch of a path we can optionally specify a list of properties to fetch.
121 * If you do not specify a list of properties ALL the properties for that bean
122 * type are fetched.
123 * </p>
124 * <p>
125 * <b>WHERE</b> <b>{list of predicates}</b>
126 * </p>
127 * <p>
128 * The list of predicates which are joined by AND OR NOT ( and ). They can
129 * include named (or positioned) bind parameters. These parameters will need to
130 * be bound by {@link Query#setParameter(String, Object)}.
131 * </p>
132 * <p>
133 * <b>ORDER BY</b> <b>{order by properties}</b>
134 * </p>
135 * <p>
136 * The list of properties to order the result. You can include ASC (ascending)
137 * and DESC (descending) in the order by clause.
138 * </p>
139 * <p>
140 * <b>LIMIT</b> <b>{max rows}</b> [ OFFSET <i>{first row}</i> ]
141 * </p>
142 * <p>
143 * The limit offset specifies the max rows and first row to fetch. The offset is
144 * optional.
145 * </p>
146 * <h4>Examples of Ebean's Query Language</h4>
147 * <p>
148 * Find orders fetching its id, shipDate and status properties. Note that the id
149 * property is always fetched even if it is not included in the list of fetch
150 * properties.
151 * </p>
152 * <pre>{@code
153 *
154 * select (shipDate, status)
155 *
156 * }</pre>
157 * <p>
158 * Find orders with a named bind variable (that will need to be bound via
159 * {@link Query#setParameter(String, Object)}).
160 * </p>
161 * <pre>{@code
162 *
163 * where customer.name like :custLike
164 *
165 * }</pre>
166 * <p>
167 * Find orders and also fetch the customer with a named bind parameter. This
168 * will fetch and populate both the order and customer objects.
169 * </p>
170 * <pre>{@code
171 *
172 * fetch customer
173 * where customer.id = :custId
174 *
175 * }</pre>
176 * <p>
177 * Find orders and also fetch the customer, customer shippingAddress, order
178 * details and related product. Note that customer and product objects will be
179 * "Partial Objects" with only some of their properties populated. The customer
180 * objects will have their id, name and shipping address populated. The product
181 * objects (associated with each order detail) will have their id, sku and name
182 * populated.
183 * </p>
184 * <pre>{@code
185 *
186 * fetch customer (name)
187 * fetch customer.shippingAddress
188 * fetch details
189 * fetch details.product (sku, name)
190 *
191 * }</pre>
192 *
193 * @param <T> the type of Entity bean this query will fetch.
194 */

258  /**
259   * Set the index(es) to search for a document store which uses partitions.
260   * <p>
261   * For example, when executing a query against ElasticSearch with daily indexes we can
262   * explicitly specify the indexes to search against.
263   * </p>
264   * <pre>{@code
265   *
266   *   // explicitly specify the indexes to search
267   *   query.setDocIndexName("logstash-2016.11.5,logstash-2016.11.6")
268   *
269   *   // search today's index
270   *   query.setDocIndexName("$today")
271   *
272   *   // search the last 3 days
273   *   query.setDocIndexName("$last-3")
274   *
275   * }</pre>
276   * <p>
277   * If the indexName is specified with ${daily} e.g. "logstash-${daily}" ... then we can use
278   * $today and $last-x as the search docIndexName like the examples below.
279   * </p>
280   * <pre>{@code
281   *
282   *   // search today's index
283   *   query.setDocIndexName("$today")
284   *
285   *   // search the last 3 days
286   *   query.setDocIndexName("$last-3")
287   *
288   * }</pre>
289   *
290   * @param indexName The index or indexes to search against
291   * @return This query
292   */
293  Query<T> setDocIndexName(String indexName);

308   * If you do not call this method on a query the "Implicit AutoTune mode" is

312   * AutoTune can add additional fetch paths to the query and specify which

332   * This means that Ebean will not add any predicates to the query for filtering out

343   * This is intended to be used when the query is not a user initiated query and instead

348  Query<T> setDisableReadAuditing();

350  /**
351   * Specify the properties to fetch on the root level entity bean in comma delimited format.
352   * <p>
353   * The Id property is automatically included in the properties to fetch unless setDistinct(true)
354   * is set on the query.
355   * </p>
356   * <p>
357   * Use {@link #fetch(String, String)} to specify specific properties to fetch
358   * on other non-root level paths of the object graph.
359   * </p>
360   * <pre>{@code
361   *
362   * List<Customer> customers =
363   *     ebeanServer.find(Customer.class)
364   *     // Only fetch the customer id, name and status.
365   *     // This is described as a "Partial Object"
366   *     .select("name, status")
367   *     .where.ilike("name", "rob%")
368   *     .findList();
369   *
370   * }</pre>
371   *
372   * @param fetchProperties the properties to fetch for this bean (* = all properties).
373   */
374  Query<T> select(String fetchProperties);
375
376  /**
377   * Specify a path to fetch eagerly including specific properties.
378   * <p>
379   * Ebean will endeavour to fetch this path using a SQL join. If Ebean determines that it can
380   * not use a SQL join (due to maxRows or because it would result in a cartesian product) Ebean
381   * will automatically convert this fetch query into a "query join" - i.e. use fetchQuery().
382   * </p>
383   * <pre>{@code
384   *
385   * // query orders...
386   * List<Order> orders =
387   *     ebeanServer.find(Order.class)
388   *       // fetch the customer...
389   *       // ... getting the customers name and phone number
390   *       .fetch("customer", "name, phoneNumber")
391   *
392   *       // ... also fetch the customers billing address (* = all properties)
393   *       .fetch("customer.billingAddress", "*")
394   *       .findList();
395   * }</pre>
396   * <p>
397   * If columns is null or "*" then all columns/properties for that path are fetched.
398   * </p>
399   * <pre>{@code
400   *
401   * // fetch customers (their id, name and status)
402   * List<Customer> customers =
403   *     ebeanServer.find(Customer.class)
404   *     .select("name, status")
405   *     .fetch("contacts", "firstName,lastName,email")
406   *     .findList();
407   *
408   * }</pre>
409   *
410   * @param path            the property path we wish to fetch eagerly.
411   * @param fetchProperties properties of the associated bean that you want to include in the
412   *                        fetch (* means all properties, null also means all properties).
413   */
414  Query<T> fetch(String path, String fetchProperties);
415
416  /**
417   * Fetch the path and properties using a "query join" (separate SQL query).
418   * <p>
419   * This is the same as:
420   * </p>
421   * <pre>{@code
422   *
423   *  fetch(path, fetchProperties, new FetchConfig().query())
424   *
425   * }</pre>
426   * <p>
427   * This would be used instead of a fetch() when we use a separate SQL query to fetch this
428   * part of the object graph rather than a SQL join.
429   * </p>
430   * <p>
431   * We might typically get a performance benefit when the path to fetch is a OneToMany
432   * or ManyToMany, the 'width' of the 'root bean' is wide and the cardinality of the many
433   * is high.
434   * </p>
435   *
436   * @param path            the property path we wish to fetch eagerly.
437   * @param fetchProperties properties of the associated bean that you want to include in the
438   *                        fetch (* means all properties, null also means all properties).
439   */
440  Query<T> fetchQuery(String path, String fetchProperties);
441
442  /**
443   * Fetch the path and properties lazily (via batch lazy loading).
444   * <p>
445   * This is the same as:
446   * </p>
447   * <pre>{@code
448   *
449   *  fetch(path, fetchProperties, new FetchConfig().lazy())
450   *
451   * }</pre>
452   * <p>
453   * The reason for using fetchLazy() is to either:
454   * </p>
455   * <ul>
456   * <li>Control/tune what is fetched as part of lazy loading</li>
457   * <li>Make use of the L2 cache, build this part of the graph from L2 cache</li>
458   * </ul>
459   *
460   * @param path            the property path we wish to fetch lazily.
461   * @param fetchProperties properties of the associated bean that you want to include in the
462   *                        fetch (* means all properties, null also means all properties).
463   */
464  Query<T> fetchLazy(String path, String fetchProperties);
465
466  /**
467   * Additionally specify a FetchConfig to use a separate query or lazy loading
468   * to load this path.
469   * <pre>{@code
470   *
471   * // fetch customers (their id, name and status)
472   * List<Customer> customers =
473   *     ebeanServer.find(Customer.class)
474   *     .select("name, status")
475   *     .fetch("contacts", "firstName,lastName,email", new FetchConfig().lazy(10))
476   *     .findList();
477   *
478   * }</pre>
479   *
480   * @param path the property path we wish to fetch eagerly.
481   */
482  Query<T> fetch(String path, String fetchProperties, FetchConfig fetchConfig);
483

485   * Specify a path to fetch eagerly including all its properties.

487   * Ebean will endeavour to fetch this path using a SQL join. If Ebean determines that it can

489   * will automatically convert this fetch query into a "query join" - i.e. use fetchQuery().
490   * </p>
491   * <pre>{@code
492   *
493   * // fetch customers (their id, name and status)
494   * List<Customer> customers =
495   *     ebeanServer.find(Customer.class)
496   *     // eager fetch the contacts
497   *     .fetch("contacts")
498   *     .findList();

500   * }</pre>

586  /**
587   * Execute the query iterating over the results.
588   * <p>
589   * Note that findIterate (and findEach and findEachWhile) uses a "per graph"
590   * persistence context scope and adjusts jdbc fetch buffer size for large
591   * queries. As such it is better to use findList for small queries.
592   * </p>
593   * <p>
594   * Remember that with {@link QueryIterator} you must call {@link QueryIterator#close()}
595   * when you have finished iterating the results (typically in a finally block).
596   * </p>
597   * <p>
598   * findEach() and findEachWhile() are preferred to findIterate() as they ensure
599   * the jdbc statement and resultSet are closed at the end of the iteration.
600   * </p>
601   * <p>
602   * This query will execute against the EbeanServer that was used to create it.
603   * </p>
604   * <pre>{@code
605   *
606   *  Query<Customer> query =
607   *    ebeanServer.find(Customer.class)
608   *     .where().eq("status", Status.NEW)
609   *     .order().asc("id");
610   *
611   *  QueryIterator<Customer> it = query.findIterate();
612   *  try {
613   *    while (it.hasNext()) {
614   *      Customer customer = it.next();
615   *      // do something with customer ...
616   *    }
617   *  } finally {
618   *    // close the underlying resources
619   *    it.close();
620   *  }
621   *
622   * }</pre>
623   */
624  QueryIterator<T> findIterate();
625
626  /**
627   * Execute the query processing the beans one at a time.
628   * <p>
629   * This method is appropriate to process very large query results as the
630   * beans are consumed one at a time and do not need to be held in memory
631   * (unlike #findList #findSet etc)
632   * </p>
633   * <p>
634   * Note that findEach (and findEachWhile and findIterate) uses a "per graph"
635   * persistence context scope and adjusts jdbc fetch buffer size for large
636   * queries. As such it is better to use findList for small queries.
637   * </p>
638   * <p>
639   * Note that internally Ebean can inform the JDBC driver that it is expecting larger
640   * resultSet and specifically for MySQL this hint is required to stop it's JDBC driver
641   * from buffering the entire resultSet. As such, for smaller resultSets findList() is
642   * generally preferable.
643   * </p>
644   * <p>
645   * Compared with #findEachWhile this will always process all the beans where as
646   * #findEachWhile provides a way to stop processing the query result early before
647   * all the beans have been read.
648   * </p>
649   * <p>
650   * This method is functionally equivalent to findIterate() but instead of using an
651   * iterator uses the QueryEachConsumer (SAM) interface which is better suited to use
652   * with Java8 closures.
653   * </p>
654   * <pre>{@code
655   *
656   *  ebeanServer.find(Customer.class)
657   *     .where().eq("status", Status.NEW)
658   *     .order().asc("id")
659   *     .findEach((Customer customer) -> {
660   *
661   *       // do something with customer
662   *       System.out.println("-- visit " + customer);
663   *     });
664   *
665   * }</pre>
666   *
667   * @param consumer the consumer used to process the queried beans.
668   */
669  void findEach(Consumer<T> consumer);
670
671  /**
672   * Execute the query using callbacks to a visitor to process the resulting
673   * beans one at a time.
674   * <p>
675   * Note that findEachWhile (and findEach and findIterate) uses a "per graph"
676   * persistence context scope and adjusts jdbc fetch buffer size for large
677   * queries. As such it is better to use findList for small queries.
678   * </p>
679   * <p>
680   * This method is functionally equivalent to findIterate() but instead of using an
681   * iterator uses the QueryEachWhileConsumer (SAM) interface which is better suited to use
682   * with Java8 closures.
683   * </p>
684   * <pre>{@code
685   *
686   *  ebeanServer.find(Customer.class)
687   *     .fetch("contacts", new FetchConfig().query(2))
688   *     .where().eq("status", Status.NEW)
689   *     .order().asc("id")
