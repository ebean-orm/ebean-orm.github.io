022/**
023 * Provides the API for fetching and saving beans to a particular database.
024 *
025 * <h5>Registration with the DB singleton</h5>
026 * <p>
027 * When a Database instance is created it can be registered with the DB
028 * singleton (see {@link DatabaseConfig#setRegister(boolean)}). The DB
029 * singleton is essentially a map of Database's that have been registered
030 * with it.
031 * </p>
032 * <p>
033 * The Database can then be retrieved later via {@link DB#byName(String)}.
034 * </p>
035 *
036 * <h5>The 'default' Database</h5>
037 * <p>
038 * One Database can be designated as the 'default' or 'primary' Database
039 * (see {@link DatabaseConfig#setDefaultServer(boolean)}. Many methods on DB
040 * such as {@link DB#find(Class)} etc are actually just a convenient way to
041 * call methods on the 'default/primary' Database.
042 * </p>
043 *
044 * <h5>Constructing a Database</h5>
045 * <p>
046 * Database's are constructed by the DatabaseFactory. They can be created
047 * programmatically via {@link DatabaseFactory#create(DatabaseConfig)} or they
048 * can be automatically constructed on demand using configuration information in
049 * the application.properties file.
050 * </p>
051 *
052 * <h5>Example: Get a Database</h5>
053 * <p>
054 * <pre>{@code
055 *
056 *   // Get access to the Human Resources Database
057 *   Database hrDatabase = DB.byName("hr");
058 *
059 *
060 *   // fetch contact 3 from the HR database
061 *   Contact contact = hrDatabase.find(Contact.class, new Integer(3));
062 *
063 *   contact.setStatus("INACTIVE"); ...
064 *
065 *   // save the contact back to the HR database
066 *   hrDatabase.save(contact);
067 *
068 * }</pre>
069 *
070 * <h5>Database vs DB API</h5>
071 * <p>
072 * Database provides additional API compared with DB. For example it
073 * provides more control over the use of Transactions that is not available in
074 * the DB API.
075 * </p>
076 *
077 * <p>
078 * <em>External Transactions:</em> If you wanted to use transactions created
079 * externally to Ebean then Database provides additional methods where you
080 * can explicitly pass a transaction (that can be created externally).
081 * </p>
082 *
083 * <p>
084 * <em>Bypass ThreadLocal Mechanism:</em> If you want to bypass the built in
085 * ThreadLocal transaction management you can use the createTransaction()
086 * method. Example: a single thread requires more than one transaction.
087 * </p>
088 *
089 * @see DB
090 * @see DatabaseFactory
091 * @see DatabaseConfig
092 */

354  /**
355   * Sort the list in memory using the sortByClause which can contain a comma delimited
356   * list of property names and keywords asc, desc, nullsHigh and nullsLow.
357   * <ul>
358   * <li>asc - ascending order (which is the default)</li>
359   * <li>desc - Descending order</li>
360   * <li>nullsHigh - Treat null values as high/large values (which is the
361   * default)</li>
362   * <li>nullsLow- Treat null values as low/very small values</li>
363   * </ul>
364   * <p>
365   * If you leave off any keywords the defaults are ascending order and treating
366   * nulls as high values.
367   * </p>

369   * Note that the sorting uses a Comparator and Collections.sort(); and does
370   * not invoke a DB query.
371   * </p>

374   *
375   *   // find orders and their customers
376   *   List<Order> list = database.find(Order.class)
377   *     .fetch("customer")
378   *     .order("id")
379   *     .findList();
380   *
381   *   // sort by customer name ascending, then by order shipDate
382   *   // ... then by the order status descending
383   *   database.sort(list, "customer.name, shipDate, status desc");
384   *
385   *   // sort by customer name descending (with nulls low)
386   *   // ... then by the order id
387   *   database.sort(list, "customer.name desc nullsLow, id");
388   *
389   * }</pre>
390   *
391   * @param list         the list of entity beans
392   * @param sortByClause the properties to sort the list by
393   */

530  /**
531   * Start a transaction with 'REQUIRED' semantics.
532   * <p>
533   * With REQUIRED semantics if an active transaction already exists that transaction will be used.
534   * </p>
535   * <p>
536   * The transaction is stored in a ThreadLocal variable and typically you only
537   * need to use the returned Transaction <em>IF</em> you wish to do things like
538   * use batch mode, change the transaction isolation level, use savepoints or
539   * log comments to the transaction log.
540   * </p>
541   * <p>
542   * Example of using a transaction to span multiple calls to find(), save()
543   * etc.
544   * </p>
545   * <p>
546   * <h3>Using try with resources</h3>
547   * <pre>{@code
548   *
549   *    // start a transaction (stored in a ThreadLocal)
550   *
551   *    try (Transaction txn = database.beginTransaction()) {
552   *
553   *        Order order = database.find(Order.class, 10);
554   *        ...
555   *        database.save(order);
556   *
557   *        txn.commit();
558   *    }
559   *
560   * }</pre>
561   * <p>
562   * <h3>Using try finally block</h3>
563   * <pre>{@code
564   *
565   *    // start a transaction (stored in a ThreadLocal)
566   *    Transaction txn = database.beginTransaction();
567   *    try {
568   *        Order order = database.find(Order.class,10);
569   *
570   *        database.save(order);
571   *
572   *        txn.commit();
573   *
574   *    } finally {
575   *        txn.end();
576   *    }
577   *
578   * }</pre>
579   * <p>
580   * <h3>Transaction options</h3>
581   * <pre>{@code
582   *
583   *     try (Transaction txn = database.beginTransaction()) {
584   *
585   *       // explicitly turn on/off JDBC batch use
586   *       txn.setBatchMode(true);
587   *       txn.setBatchSize(50);
588   *
589   *       // control flushing when mixing save and queries
590   *       txn.setBatchFlushOnQuery(false);
591   *
592   *       // turn off persist cascade if needed
593   *       txn.setPersistCascade(false);
594   *
595   *       // for large batch insert processing when we do not
596   *       // ... need the generatedKeys, don't get them
597   *       txn.setBatchGetGeneratedKeys(false);
598   *
599   *       // explicitly flush the JDBC batch buffer
600   *       txn.flush();
601   *
602   *       ...
603   *
604   *       txn.commit();
605   *    }
606   *
607   * }</pre>
608   * <p>
609   * <p>
610   * If you want to externalise the transaction management then you use
611   * createTransaction() and pass the transaction around to the various methods on
612   * Database yourself.
613   * </p>
614   */
615  Transaction beginTransaction();

622  /**
623   * Start a transaction typically specifying REQUIRES_NEW or REQUIRED semantics.
624   * <p>
625   * <p>
626   * Note that this provides an try finally alternative to using {@link #executeCall(TxScope, Callable)} or
627   * {@link #execute(TxScope, Runnable)}.
628   * </p>
629   * <p>
630   * <h3>REQUIRES_NEW example:</h3>
631   * <pre>{@code
632   * // Start a new transaction. If there is a current transaction
633   * // suspend it until this transaction ends
634   * try (Transaction txn = database.beginTransaction(TxScope.requiresNew())) {
635   *
636   *   ...
637   *
638   *   // commit the transaction
639   *   txn.commit();
640   *
641   *   // At end this transaction will:
642   *   //  A) will rollback transaction if it has not been committed
643   *   //  B) will restore a previously suspended transaction
644   * }
645   *
646   * }</pre>
647   * <p>
648   * <h3>REQUIRED example:</h3>
649   * <pre>{@code
650   *
651   * // start a new transaction if there is not a current transaction
652   * try (Transaction txn = database.beginTransaction(TxScope.required())) {
653   *
654   *   ...
655   *
656   *   // commit the transaction if it was created or
657   *   // do nothing if there was already a current transaction
658   *   txn.commit();
659   * }
660   *
661   * }</pre>
662   */
663  Transaction beginTransaction(TxScope scope);

696   * <p>
697   * Useful to put in a finally block to ensure the transaction is ended, rather
698   * than a rollbackTransaction() in each catch block.
699   * </p>
700   * <p>
701   * Code example:
702   * <p>
703   * <pre>{@code
704   *
705   *   database.beginTransaction();
706   *   try {
707   *     // do some fetching and or persisting ...
